## 2. イベントシステム（部分実装・要検証）

### 現状
- `EventData`, `EventChoiceData` リソースクラス定義済み
- `EventManager` Autoload で `settlement` / `faction` / `character` / `travel` 4カテゴリからロード
- `event_screen.gd` で選択肢表示・効果適用は動作
- `QuestManager` で遅延ペイロード（`payload_only`）の仕組みは実装済み

### 改善・検証項目
- [ ] **2-1. 後続イベント予約の動作検証**
  - `QuestManager.record_outcome()` → `_armed` への装填フロー確認
  - 現在 `outcome == &"poison"` のときのみペイロード装填のハードコード
  - 汎用的な後続イベント予約（他のクエストタイプ）に対応できていない
  - `payload_min_gap` ノード経過後に `get_pending_payload()` で正しく発火するか検証
- [ ] **2-2. QuestData の実データ確認**
  - `resources/quests/` にどれだけの `.tres` が存在するか
  - クエスト定義と実イベントとの整合（`payload_event` のIDが実在するか）
- [ ] **2-3. イベント数の充実度確認**
  - 各カテゴリ（settlement / faction / character / travel）のイベント数
  - Act別のフィルタリング（`required_act`）が機能しているか
  - 各キャラクター固有イベント（`required_character`）の充実度
- [ ] **2-4. sets_flag の消費側確認**
  - `EventChoiceData.sets_flag` で立てたフラグが後続の `requirement` で参照されているか
  - `requirement` パーサー（`_check_requirement`）に `flag==` 系の条件が未実装
  - → フラグを立てても後続で参照できない可能性
- [ ] **2-5. イベント戦闘のバリエーション**
  - `triggers_combat` 時の敵選出が「ランダム1体」のみ
  - イベント固有の敵（特定IDの敵と戦う）の指定不可
  - → `EventChoiceData` に `combat_enemy_id` フィールド追加を検討
- [ ] **2-6. companion_id による同行者加入の動作確認**
  - `CompanionDatabase.get_companion()` が正しく返すか
  - 既に同行者がいる場合の入れ替え処理の有無

| 優先度 | 項目 | 理由 |
|--------|------|------|
| **高** | 1. アイテムシステム | ゲームの深み・リプレイ性に直結。レリック＝ランの個性 |
| **高** | 3. フレーバーテキスト表示 | カード体験の質。カード拡大ポップアップも合わせて |
| **高** | 4-1. エディターのダブルクリック | 開発効率に直結。修正は小規模 |
| **中** | 2-1〜2-4. イベント検証・改善 | 既存機能の品質保証。フラグ参照の抜けは実質バグ |
| **中** | 5-1. デッキブラウズ | プレイヤーの意思決定に必要な情報アクセス |
| **中** | 5-3. バイク耐久度 | データ・UIが揃っているのに未接続 |
| **中** | 5-9. 強化詳細表示 | 情報不足による不満の原因 |
| **低** | 5-2. 看破機能TODO | ゲームプレイへの影響は限定的 |
| **低** | 5-4. 医薬品の戦闘外使用 | アイテムシステムに統合して実装 |
