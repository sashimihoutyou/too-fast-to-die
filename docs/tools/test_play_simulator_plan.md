# 擬似テストプレイツール企画案

## 目的

`tres` ファイルに定義されたキャラクター・初期デッキ・カード・敵データを読み込み、同じ入力条件で何度も疑似ランを実行して、ゲームバランスとデータ不整合を早期に検出する。

## MVPで検証する指標

- ラン数、クリア数、クリア率
- 平均到達Act、平均撃破敵数、平均残HP
- 初期デッキごとの勝率差
- 戦闘ごとの平均ターン数、被ダメージ、使用カード回数
- データ検証エラー数（存在しないカードID、未対応status_effect、強化値が実質no-opなど）

## 入力データ

- `resources/characters/*.tres`: キャラクター能力、初期デッキID、固有システム
- `resources/cards/**/*.tres`: カード効果、コスト、強化値、状態異常、使用不可カード
- `resources/enemies/**/*.tres`: Act別の敵、ボス、弱点、カテゴリ
- 任意のシミュレーション設定: ラン数、乱数seed、対象キャラクター、Act数、報酬選択ポリシー

## アーキテクチャ方針

1. **データ監査レイヤー**
   - GodotのResourceLoaderで`tres`を読む。
   - 初期デッキIDがCardDatabaseに存在するか検証する。
   - `status_effect` が `CombatManager` の対応済み効果に含まれるか検証する。
   - 強化済みプレビューが空になるカードを検出する。
2. **戦闘シミュレーションレイヤー**
   - 既存の `DeckManager` / `CombatManager` を可能な限り使う。
   - UIに依存せず、カード選択AIが `can_play_card()` と `preview_damage()` を見て操作する。
   - まずは「倒せる敵を優先」「ブロック不足時は防御優先」の決定的ヒューリスティックにする。
3. **ラン進行レイヤー**
   - Actごとの通常敵、エリート、ボスをEnemyDatabaseから選ぶ。
   - 報酬カードはポリシーで選ぶ（例: 高ダメージ優先、デッキ枚数上限、キャラ専用優先）。
   - 休息はHP閾値で休息/強化を選ぶ。
4. **レポートレイヤー**
   - JSONとMarkdownの両方を出力する。
   - seedと意思決定ログを保存し、異常ランを再現できるようにする。

## CLI案

```bash
godot --headless --path . --script res://tools/sim/run_simulation.gd -- --character ex_raider --runs 100 --seed 12345 --report reports/sim_ex_raider.md
```

## 実装ステップ

1. `tools/sim/data_audit.gd` を作り、Resource読み込みとデータ整合チェックだけを行う。
2. `tools/sim/battle_agent.gd` を作り、1戦だけを自動操作して勝敗とログを返す。
3. `tools/sim/run_simulation.gd` で複数戦・複数ランをつなぐ。
4. CIで少数seed（例: 10ラン）を回し、クラッシュやデータ不整合を検出する。
5. バランス調整用に100〜1000ランのローカル実行プリセットを用意する。

## 初期の合格基準案

- 全キャラクターの初期デッキでデータ監査エラーが0件。
- 10ランのスモークテストでクラッシュが0件。
- 各キャラクターのAct1突破率が極端に低すぎない（目安: 30%以上）。
- 同一seedでは同一ログと同一結果になる。

## 注意点

- UIシーンを直接操作するテストではなく、AutoloadとResourceを使ったロジックテストを優先する。
- AIが下手すぎるとカードや敵の評価が歪むため、勝率は絶対値よりも変更前後の差分を見る。
- 疑似プレイではイベントやショップの選択を簡略化し、戦闘バランスの検証から始める。
